HTTP

超文本传输协议HTTP的定义

http(hypertext transfer protocol)是web系统最核心的内容，是web服务器和客户端直接进行数据传输的规则。
http协议是用于从www服务器传输超文本到本地浏览器的传送协议，不仅保证计算机正确快速的传输超文本文档，还确定传输文档中的哪一部分，以及哪部分内容最先显示等。http是一个应用层协议，由请求和相应构成，是一个标准的客户端模型。

HTTP的特点

1. 支持客户端/服务器模式，支持基本认证和安全认证
2. 简单快速，客户端向服务器发送请求时，只需传送请求方法和路径。请求方法常用的有GET,HEAD,POST(区分大小写且可自定义请求方法)。每种方法规定了客户端与服务器联系的类型不同。
3. 灵活，http允许传输任意类型的数据对象。正在传输的类型由content-type加以标记
4. http0.9和1.0使用非持续连接，即限制每次连接只处理一个请求，服务器处理完客户的请求，并收到客户的应答后才断开连接。http1.1使用持续连接，即不必为每个web对象创建一个新的连接，一个连接可以传送多个对象。
5. http是无状态协议，即对于处理事务没有记忆能力，缺少状态一位置如果后续处理需要前面的信息则它必须重传，这可能导致每次连接传送的数据量增大。

HTTP的请求方法

1. OPTION：范湖服务器针对特定资源所支持的HTTP请求方法，也可以利用向web服务器发送"*"的请求来测试服务器的功能性。
2. GET：向特定的资源发出请求。
3. HEAD：向服务器索要与GET请求相一致的响应，只不过响应体或许不会被返回。这一方法可以在不必传输整个响应内容的情况下，就可以获取包含在响应消息中的元信息。
4. POST：向指定资源提交数据进行处理请求(比如提交表单或者上传文件)，数据被包含在请求体中。POST请求可能会导致新的资源的建立或者已有资源的修改。
5. PUT：向指定资源位置上传其最新内容。
6. DELETE：请求服务器删除Request-URI所标识的资源。
7. TRACE：回显服务器收到的请求，主要用于测试或者诊断。
8. CONNECT： HTTP/1.1协议中预留给能给够将连接改为管道方式的代理服务器。
大部分情况下只会用到GET和POST方法，这些方法区分大小写而且可以自定义请求方法。当某个请求所针对的资源不支持对应的请求方法的史诗，服务器应当返回状态码405，当服务器不认识或者不支持对应的请求方法的时候，应返回状态码501。

HTTP的工作流程

http是一个无状态的协议。这意味着当一个客户端向服务器发送请求，然后服务器返回响应后连接就被关闭了，在服务器端不保留连接有关信息。http遵循请求/应答模型，客户端向服务器发送请求，服务器处理请求并返回适当的应答。所有http连接都被构造成一套请求和应答，该过程经历四个阶段，分别是建立连接，发送请求信息，发送响应信息，关闭连接，如下所示：
    1. 客户端通过TCP三次握手与服务器建立连接
    2. TCP建立连接成功后，向服务器发送HTTP请求
    3. 服务器接收客户端的HTTP请求后，将返回应答，并向客户端发送数据包
    4. 客户端通过TCP四次断开，与服务器断开TCP连接

HTTP的报文格式

http的报文分为请求报文和应答报文

HTTP请求报文

> http请求报文格式


第一行为”请求行”，第二三四行为”请求头部”，第五行为请求空行，第六行为”请求正文”
1. 请求行：由请求方法，URL，协议版本构成，中间用空格符分割，请求方法包括GET,POST等。协议版主要有HTTP/1.0和HTTP/1.1。
2. 请求头部包含很多客户端环境以及请求正文的有用信息。由”关键字:值”组成
3. 空行，必不可少，表示请求头部结束，下面就是请求正文
4. 请求正文：可选部分，GET就没有请求正文，POST可以提交表单，数据或者上传文件。

> http响应报文格式


第一行为”状态行”，第二三四行为”响应头部”，第五行为行为空行，第六行为”响应正文”
1. 状态行有3部分构成，分别是协议版本，状态码和状态码的描述，中间用空格隔开。状态码为3位数字，200-299状态码表示成功(如200表示响应成功)，300-399状态码表示资源重定向(如302表示资源重定向)，400-499状态码表示客户端请求出错(如404表示请求资源不存在)，500-599状态码表示服务器端请求出错(如500表示服务器内部错误)。
2. 响应头部包含很多客户端环境以及请求正文的有用信息。由”关键字:值”组成
3. 空行，必不可少，表示响应头部结束，下面就是响应正文
4. 响应正文，服务器返回的文档，通常为HTML网页

HTTP请求走私

HTTP请求走私是一种干扰网站处理从一个或多个用户接收的HTTP请求序列的方式的技术。使攻击者可以绕过安全控制，未经授权访问敏感数据并危害其他应用程序用户。

HTTP请求走私的成因

前端服务器(CDN)和后端服务器接收的数据不同步，引起对客户传入的数据理解不一致，从而导致漏洞的产生。
大多数HTTP请求走私漏洞的出现是因为HTTP规范提供了两种不同的方法来指定请求的结束位置：Content-Length标头和Transfer-Encoding标头。同时使用这两种不同的方法的时候，Content-Length就无效，而当使用多个服务器时，对客户端传入的数据理解不一致时就会出现有些服务器认为Content-Length的长度有效，有些认为Transfer-Encoding有效。而一般情况下，反向代理服务器与后端的源站度武器之间会重用TCP连接，这样长度就会拼接到下一次请求进行请求，从而导致HTTP请求走私。
【RFC2616规范：如果接收的消息同时包含传输编码头字段(Transfer-Encoding)和内容长度头(Content-Length)字段，则必须忽略后者。】

HTTP请求走私实现

当向代理服务器发送一个比较模糊的HTTP请求的时候，由于两个服务器的实现方式不同，代理服务器认为这是一个HTTP请求，然后将其转发给了后端的源站服务器，但源站服务器经过解析处理后，只认为其中的一部分是正常请求，另一部分是走私请求，当该走私请求部分对正常用户的请求造成影响后，就实现了HTTP请求走私。完成此操作的确切方式取决于两个服务器的行为：
    CL-TE：前端服务器使用Content-Length标头，而后端服务器使用Transfer-Encoding标头
    TE-CL：前端服务器使用Transfer-Encoding标头，而后端服务器使用Content-Length标头
    TE-TE：前端服务器和后端服务器都支持Transfer-Encoding标头，但是可以通过对标头进行某种方式的混淆来诱导其中一台服务器不对其进行处理。
